◆ CommandJog

org.openpnp.spi.MotionPlanner.CompletionType.CommandJog

The motion plan is sent to the drivers, assuming an unfinished motion sequence (e.g. when Jogging). More specifically, the motion planner's deceleration to still-stand will not be enforced, it is entirely up to the controller. If a subsequent motion command arrives soon enough, there might be no (or less) deceleration between the moves.
If the driver supports asynchronous execution, this does not wait for the driver to physically complete.

◆ CommandStillstand

org.openpnp.spi.MotionPlanner.CompletionType.CommandStillstand

The motion plan is sent to the drivers, finishing in still-stand.
If the driver supports asynchronous execution, this does not wait for the driver to physically complete.

◆ WaitForStillstand

org.openpnp.spi.MotionPlanner.CompletionType.WaitForStillstand

The motion plan is executed with the drivers, finishing in still-stand.
This does always wait for the driver to complete i.e. the caller can be sure the machine has physically arrived at the final location.

◆ WaitForStillstandIndefinitely

org.openpnp.spi.MotionPlanner.CompletionType.WaitForStillstandIndefinitely

Like WaitForFullCoordination but wait "forever" i.e. with very long timeout. Used e.g. for homing.

◆ WaitForUnconditionalCoordination

org.openpnp.spi.MotionPlanner.CompletionType.WaitForUnconditionalCoordination

Like WaitForStillStand but the wait will also be done, if no motion is thought to be pending (used when it is though that the machine might have moved through custom Actuator Gcode, the ContactProbeNozzle Probe actuator is one example).
